# Пример: Audio Transcription Service

Этот пример демонстрирует **итеративный процесс** создания документации.

## Процесс создания

### 1. Discovery

См. [docs/_discovery-log.md](docs/_discovery-log.md)

- Интервью с заказчиком
- Выявление реальных потребностей
- Сужение scope на основе ответов

### 2. PRD черновик

См. [docs/_prd-v1-draft.md](docs/_prd-v1-draft.md)

- Первая версия с широким scope
- Feedback заказчика: "batch не нужен для MVP"
- Итерация на основе обратной связи

### 3. PRD финальный

См. [docs/PRD.md](docs/PRD.md)

- Scope сужен для MVP после валидации
- Цели согласованы с заказчиком

> **Примечание:** PRD.md в этом примере показывает **расширенную версию** (v1.0), куда вошли функции из Future Scope (batch, SRT). Это демонстрирует, что scope может расширяться после успешного MVP.

### 4. SPEC

См. [docs/SPEC.md](docs/SPEC.md)

- AC согласованы с заказчиком
- Каждое требование проверяемо

### 5. ADR

См. [docs/decisions/](docs/decisions/)

- Технологии выбраны после вопроса о предпочтениях
- Решения обоснованы

### 6. CLAUDE.md

См. [CLAUDE.md](CLAUDE.md)

- Создан **последним**, когда все решения приняты
- Технологический стек соответствует ADR
- Ключевые решения ссылаются на ADR

## Ключевые уроки

1. **Discovery первичен** — не начинай писать PRD без интервью
2. **Validation loops** — показывай промежуточные результаты
3. **MVP scope** — сужай, пока не останется минимум
4. **CLAUDE.md последним** — он агрегирует решения из ADR
